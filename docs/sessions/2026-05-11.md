## Implementation work landed

### `[x]` AD-31.1: privilege drop in semadrawd

Done and bench-verified. Three commits on origin:

  - `aa88638` initial implementation
  - `a731889` first compile-fix attempt (broken, see below)
  - `4088345` final compile-fix (drop gid-side verification)

Three files changed:

  - `semadraw/src/daemon/semadrawd.zig`: adds `dropPrivileges()`
    function (~80 lines) before `main()`, with a call site between
    `daemon.initCompositor()` and `daemon.run()`. Reads
    `SEMADRAW_RUN_UID` and `SEMADRAW_RUN_GID` from environment via
    `posix.getenv()`, validates (refuses uid 0 / gid 0 explicitly,
    refuses missing or invalid env vars when running as root), calls
    `posix.setgid(target_gid)` then `posix.setuid(target_uid)`,
    verifies the drop with `posix.getuid()` / `posix.geteuid()`. The
    gid-side verification was dropped during the Zig 0.15.2 fix
    iteration; see "What went wrong with the Zig fix" below for the
    full story.
  - `install.sh`: idempotent `_semadraw` user/group creation after
    the `/boot/loader.conf` section. Uses `pw groupadd` / `pw
    useradd` with `-d /nonexistent`, `-s /usr/sbin/nologin`. UID/GID
    are not pinned; `pw` picks the next available system-account
    id.
  - `s6/utf/semadrawd/run`: discovers `_semadraw` uid/gid via
    `id -u _semadraw` / `id -g _semadraw` and exports
    `SEMADRAW_RUN_UID` / `SEMADRAW_RUN_GID` before `exec`'ing
    semadrawd.

Bench verification chain (all four passed):

  - `ps -o uid,user,command -ax | grep semadrawd` shows uid 1002
    (`_semadraw`), not 0. Privilege drop took.
  - `sudo procstat -e $(pgrep semadrawd)` shows
    `SEMADRAW_RUN_UID=1002` and `SEMADRAW_RUN_GID=1002` in the
    daemon's environment, confirming the s6 run script's export
    chain works.
  - `procstat -e` without sudo returns empty environment, which is
    itself a small validation: before AD-31.1 any user could read
    semadrawd's environment because the daemon ran as root; now
    the privilege boundary is real enough that environment-reading
    requires elevated privilege.
  - `semadraw-term` connects, creates a 640x400 surface, starts a
    session. Client connectivity preserved across the privilege
    drop. The retained-fd-across-setuid pattern works exactly as
    ADR 0006 §1 specified.

### What went wrong with the Zig fix

The initial implementation called `posix.getgid()` and
`posix.getegid()` for the drop-verification step. Zig 0.15.2's
`std.posix` does not expose those functions: it exposes `getuid`,
`geteuid`, `setuid`, `seteuid`, `setgid`, `setegid`, but not
`getgid` or `getegid`. The first compile attempt failed at line
2419 with "struct 'posix' has no member named 'getgid'".

The first fix attempt substituted `std.c.getgid()` / `std.c.getegid()`
on the assumption that since the binary links libc, the C library
wrappers would be exposed. They are not: `std.c` exposes `setuid`,
`setgid`, `getuid`, `geteuid` but again no `getgid` / `getegid`.
Second compile attempt failed with "struct 'c' has no member named
'getgid'".

The final fix dropped the gid-side verification entirely. The
verify-after-drop step was defense-in-depth against a hypothetical
kernel bug that returns success from `setgid` without actually
changing the gid; if `posix.setgid` returns success and the kernel
does not provide an exposed getter for gid, there is no available
API to perform the check, so the gid half of the verification is
removed. The uid-side verification (which catches the plausible
failure mode, operator misconfiguration of `SEMADRAW_RUN_UID`)
remains in place.

The lesson worth recording for future Zig work: when the sandbox
cannot compile-test Zig, "I think `std.X` exposes Y" is a guess.
The cost of asking the operator to grep the actual stdlib is tiny
compared to the cost of two failed iterations. The pattern is now
"verify the stdlib surface first, write the code against verified
API, then ship."

## Architectural decisions reached

### Audio: discovery document and three ADRs

The audiofs design has gone from "open-ended exploration with six
unresolved Q1-Q6 questions" to "tractable sequence with the three
largest decisions resolved."

**`audiofs/docs/audio-design-space.md` (commit `3af421f`).** 797-line
discovery document modelled on `docs/Thoughts.md` (chronofs's
discovery doc). Surveys eight architectural models: OSS, ALSA,
PulseAudio, PipeWire, JACK, sndio, Plan 9, the existing audiofs
proposal. Each described in terms of where authoritative state lives,
where mixing happens, how the clock is handled, how it relates to
UTF's discipline. Three converging conclusions:

  - The clock-writer responsibility is the only load-bearing
    kernel-side requirement; everything else about audio could in
    principle live in userland.
  - The mixer location (Q2) is the structural fork that shapes Q1,
    Q4, and the userland architecture. Not just one of six
    independent questions.
  - sndio is more relevant as a reference architecture than the
    proposal had acknowledged. Its small API, privilege-separated
    design, and hardware-or-broker transparency are real candidate
    features.

The discovery document recommends modifying the audiofs ADR plan
from six ADRs to eight: pull the clock writer out of the implicit
framing as its own ADR, promote Q2 (substrate-vs-broker) to the
lead position, add a userland-architecture ADR.

**`audiofs/docs/adr/0003-clock-writer.md` (commit `26be39e`).**
Accepted. Commits audiofs to writing the chronofs clock region from
kernel context, regardless of how the larger architecture resolves.
Key load-bearing decision: `samples_written` represents frames the
codec has actually clocked out (read from `snd(4)`'s buffer-pointer
mechanism via `xxxchannel_getptr()`), not frames-handed-to-OSS
(what semaaud counts today). This is a precision improvement; the
wire format does not change but the meaning of the number subtly
does. Explicitly independent of Q2 (mixer location) and Q3 (OSS
coexistence, ADR 0002).

**`audiofs/docs/adr/0004-mixer-location.md` (commit `a81d833`).**
Accepted. Resolves Q2 as Option A (single-writer): semasound is
the sole writer to audiofs's audio data path; the kernel substrate
sees mixed output, not per-stream input. The decision is recorded
as being on architectural-quality grounds rather than
architectural-discipline grounds: the discipline argument slightly
favours Option B (multi-writer), but four practical observations
tip toward Option A. The ADR explicitly lists what would change
the decision (a concrete consumer of substrate-level per-stream
state), so the door stays open without weakening the current
commitment.

**`audiofs/docs/adr/0005-userland-architecture.md` (commit
`1e75e60`).** Accepted. Specifies semasound's architectural shape.
Surfaces the gap between the audiofs proposal's framing
("semasound = semaaud minus OSS plus audiofs") and the reality:
semaaud today uses preemption (one client per target,
policy-driven displacement), not mixing. semasound as committed
by the audiofs proposal requires true multi-client mixing within
a target, which is genuinely new behaviour. The ADR specifies
five sub-decisions: per-target listening sockets that accept
multiple connections, per-connection session structure, per-target
mix workers, policy applied at connection and preemption events
(default behaviour changes from preemption to additive mixing),
semasound with no kernel-side responsibilities.

### Verify before guessing

The AD-31.1 Zig iteration sequence (three attempts for the same
fix) was a clear example of the failure mode of guessing at API I
cannot see. The cost of asking the operator to run two `grep`
commands against the bench's stdlib was minutes; the cost of two

workflow.

  - First request the relevant stdlib surface via `grep`.
  - Write code against verified API only.
  - Mark uncompiled code explicitly as "needs bench verification"
    rather than implying readiness.

### `[x]` AD-31.2: peer-uid identification

Commit `57311e9`. Done. Adds `semadraw/src/daemon/privilege.zig` (88
lines): `NOBODY_UID` / `NOBODY_GID` sentinels (65534, FreeBSD nobody
by passwd convention), `PeerCredentials` struct, an extern
declaration of `getpeereid(3)` (neither `std.posix` nor `std.c`
exposes it on Zig 0.15.2), and `getPeerCredentials(fd)` returning
`!PeerCredentials`. `ClientSession` and `RemoteSession` gain
`peer_uid` and `peer_gid` fields, populated at session-init time
from getpeereid (Unix) or set to `NOBODY_UID` (TCP). The Unix
listener's `handleNewConnection` calls `getPeerCredentials` before
the session-create call; failure closes the fd without sending a
reply (per ADR 0006 §2), and that path also fixed a pre-existing
fd-leak in the same handler. Bench-verified: client log shows
`peer uid=1001 gid=1001` for vic-run clients.

### `[x]` AD-16.6: deleteChars overlapping-shift fix

Commit `c5fb11e`. Done. A pre-existing bug in semadraw-term's
screen.zig (`deleteChars`) used `@memcpy` for the leftward shift;
when `chars_to_delete > 0` and `chars_to_shift > chars_to_delete`,
the source and destination slices overlapped and Zig's `@memcpy`
panicked with "arguments alias". Triggered in practice by running
`ls` in semadraw-term during AD-31.1 bench testing. Fix replaces
`@memcpy` with a manual forward loop (safe for left-shift overlap;
symmetric with `insertChars`'s backward loop from AD-16.4). Added
two new test cases (overlapping shift + n=0 no-op). Filed as a
sub-stage under the closed AD-16 entry, matching the pattern of
"latent edge-case bug discovered after closure."

### `[x]` AD-31.1 follow-up: inputfs publication file perms

Commit `cb71b03`. Done. Surfaced during AD-31.1 bench testing
when the cursor sprite stopped moving despite the trackpad
working. Root cause: semadrawd's `pumpCursorPosition` lazy-opens
`/var/run/sema/input/state` on every call (so the daemon can come
up before inputfs is loaded); per inputfs ADR 0013, those files
are published `root:wheel:0600`; post-AD-31.1 `_semadraw` cannot
open them.

Fix uses ADR 0013's deliberate operator escape hatch: `install.sh`
adds `hw.inputfs.dev_gid=<_semadraw gid>` and
`hw.inputfs.dev_mode=0640` to `/boot/loader.conf` (idempotent),
sets the live sysctls, and `chgrp _semadraw + chmod 0640` on the
existing `/var/run/sema/input/{state,events,focus}` files so the
running inputfs picks up the change without a reboot. ADR 0006
implementation status updated to note the follow-up. Bench-verified:
mouse and trackball cursor now move in both framebuffer and
semadraw-term display modes.

### `[x]` AD-26.1: route framebuffer-size query through daemon IPC

Commit `9be87a7`. Done. AD-26 (2026-05-10) implemented
`semadraw-term --fullscreen` by opening `/dev/draw` and calling
`DRAWFSGIOC_GET_EFIFB_INFO` directly. AD-26's closure note already
acknowledged the root-only limitation; AD-31.1's privilege drop
formalised it: per ADR 0006 §5, `/dev/draw` is restricted to the
daemon, and clients running as the operator cannot open it.

Fix routes the query through a new protocol message pair:

  - `shared/protocol_constants.json`: adds
    `OUTPUT_INFO_REQUEST=0x0060` and `OUTPUT_INFO_REPLY=0x8060`
    to the `semadraw_ipc` section (regenerated via
    `gen_constants.py`).
  - `semadraw/src/ipc/protocol.zig`: enum auto-extended; hand-added
    `OutputInfoRequestMsg` (4 bytes: output_id) and
    `OutputInfoReplyMsg` (12 bytes: output_id + width + height)
    struct types with round-trip tests.
  - `semadraw/src/daemon/semadrawd.zig`: `handleOutputInfoRequest`
    (Unix) and `handleRemoteOutputInfoRequest` (TCP) read from
    `Compositor.outputDimensions()` and reply with
    `OutputInfoReplyMsg`, or send `ERROR_REPLY` with
    `validation_failed` for unknown output_id / no output
    initialised.
  - `semadraw/src/client/connection.zig` and
    `remote_connection.zig`: `queryOutputInfo(output_id)` method
    that sends the request and waits for the matching reply.
  - `semadraw/src/apps/term/main.zig`: removes `DRAWFS_DEV`,
    `EfifbInfoIoctl`, `DRAWFSGIOC_GET_EFIFB_INFO`, the `iocOut`
    helper, and the `extern fn ioctl` declaration. New
    `queryDisplaySizeViaIpc(conn, ...)` function calls
    `conn.queryOutputInfo(0)`. `run()` reordered so
    `client.connect` happens before the size-determination block.

Bench-verified: `semadraw-term --fullscreen` as vic (uid 1001)
produces a 3840×2160 surface; the previous "queryDisplaySize

### `[x]` AD-31.3: privileged-client recognition and surface owner tagging

## What went wrong (continuation)

### Wholesale-cp apply scripts are fragile

Two failures from the continuation tonight, both rooted in the
same pattern: my apply scripts use `cp "$SRC" "$DST"` to overwrite
files wholesale, and my staged source files are baked at the
moment I produce them rather than reconstructed against the
latest tarball at apply time.

The AD-31.1-follow-up apply script overwrote `BACKLOG.md` with a
version that pre-dated `c5fb11e`'s AD-16.6 entry, silently
dropping that entry. The AD-26.1 apply found and restored it.

A later AD-26.1 attempt on Vic's `pgsd-dev` workstation left
`main.zig` apparently unchanged (the file on disk had the old
warning text and old function name) even though the commit
message claimed AD-26.1. The cause traced through several
incorrect hypotheses (stale binary, build cache, env-var typo)
before resolving: the AD-26.1 changes were never committed on
that workstation's tree. The apply had run, the `cp` had landed
the new files, but the `git commit` that followed only picked up
some of them; the working tree as captured for the next tarball
upload didn't include the `main.zig` change. The fix was a
straightforward `git push` from the workstation tree where the
change actually existed.

The lesson: **surgical patches (str_replace-style or `patch`-style
edits) are safer than wholesale `cp` for files that already exist
in the tree.** Wholesale `cp` is only safe when the staged source
is reconstructed against the live tree state at apply time, not
baked into the staged file at production time. The two-tree
gotcha (`pgsd-dev` vs `pgsd-bare-metal-test-machine`) makes this
worse because the tarball Claude sees may not match either bench
state in subtle ways.

For the next session: prefer apply scripts that use `str_replace`
or `patch` rather than `cp`, even when more verbose; establish
ground truth (sha256 + git log) on each upload before producing
new patches; suggest `clean.sh` early when build state looks
suspicious; recognize the "build output is empty / build is a
cache hit" pattern as a signal that source-on-disk may not be
what's expected.

### The terminal wrap that ate a typo

While diagnosing the AD-26.1 stale-binary mystery, I suggested
`sudo rm -rf zig-out .zig-cache` to clean Vic's build state. The
command got copy-pasted with a terminal line-wrap that split
"zig-out" into "zig-ou\nt", and the shell parsed "zig-ou" and
separately "t" as two non-existent paths that `rm -rf`
accepted silently. The cache was wiped but `zig-out/` was not.
Subsequent `zig build` then reused the old `zig-out/bin/`
artifacts as cache hits.

The right tool was `clean.sh --force`, which the project already
ships. Vic flagged this directly: "we have a `clean.sh` that we
can use." Using project tooling over hand-rolled equivalents
avoids exactly this class of typo-cascade failure. Recorded.

### Apply-script post-state anchors

The post-state grep checks in apply scripts are meant to catch
"apply did not land as expected." Several times tonight the
anchors I chose were partial-match prone (e.g. just `isPrivilegedUid`
would match a TODO comment in the pre-apply file). The fix is to
use anchors that are unique to the post-apply state: function
signatures (`pub fn isPrivilegedUid`) rather than bare names,
specific struct fields (`owner_uid: posix.uid_t`) rather than
ambiguous prefixes.

Apply-script post-state anchors must be:

  - Single-line (no wrap)
  - ≤6 words
  - Unique to the post-apply state (not present anywhere
    pre-apply, including comments)
  - Specific (function signatures, field declarations) not
    generic (bare identifiers)

### `[x]` AD-31.4 part A: devfs rule for /dev/draw

Commit `1546ec5`. Done. ADR 0006 §5 specifies a devfs rule
restricting `/dev/draw` to `_semadraw:0660` so semadrawd is
the sole userspace gatekeeper of the framebuffer. install.sh
gained a "Configuring devfs rules" block that:

  - Writes a markered managed region into `/etc/devfs.rules`
    defining `[utf_devices=10]` with
    `add path 'draw' mode 0660 group _semadraw`.
    Idempotent: re-running install detects and replaces the
    markered region.
  - Handles `devfs_system_ruleset` in rc.conf gracefully via
    a three-case `case` statement on the value `sysrc -n`
    returns: empty -> set + apply live; already `utf_devices`
    -> no-op + apply live; pointing elsewhere (operator has
    their own ruleset, e.g. GhostBSD's `devfsrules_common`)
    -> refuse to override, leave the operator's setting
    alone, print a detailed NOTICE with two
    manual-integration options. The `utf_devices` block is
    written to `/etc/devfs.rules` regardless so the operator
    has the canonical text on hand if they want to merge.
  - Applies live via `service devfs restart` only when the
    install actually set or confirmed the ruleset.
    `APPLY_LIVE` flag guards this.
  - Uninstall mirrors: removes the markered region; unsets
    `devfs_system_ruleset` only if it still points to
    `utf_devices`; runs `service devfs restart` to revert
    live.

Vic's initial inclination was to add the entry to the
operator's existing `devfsrules_common=7` ruleset (the
GhostBSD desktop convention), then withdrew that suggestion
in favour of following the ADR as written. The graceful
posture for case 3 (operator-has-other-ruleset) captures the
underlying concern without amending the ADR: UTF doesn't
clobber operator customisation, the operator gets a clear
path to integrate, and the default-perms of `/dev/draw`
(`root:wheel:0600`) keep UTF functional as defence-in-depth
rather than a load-bearing requirement.

Bench-verified: `/dev/draw` shows `crw-rw---- root:_semadraw`
after install, `sudo -u nobody cat /dev/draw` returns
EACCES, semadraw-term continues to work because semadrawd
opens `/dev/draw` as root before dropping to `_semadraw`
(the fd survives the drop).

### `[x]` AD-31.4 part B: peer_uid and owner_uid in event schema

Commit `b3432b0`. Done. ADR 0006 §9 specifies that the
unified-schema events gain uid fields for attribution.
events.zig and semadrawd.zig updated:

  - **events.zig**: four emit functions gain a uid
    parameter. `emitClientConnected` and
    `emitClientDisconnected` take
    `peer_uid: std.posix.uid_t`; `emitSurfaceCreated` and
    `emitSurfaceDestroyed` take
    `owner_uid: std.posix.uid_t`. Each function's payload
    string adds `,"peer_uid":N` or `,"owner_uid":N`. Buffer
    sizes bumped to accommodate the new field widths
    (64 -> 128, 128 -> 160, 96 -> 128, 64 -> 96).
  - **semadrawd.zig**: all 8 emit call sites updated. The
    two `emitClientConnected` sites pass `session.peer_uid`;
    the two `emitSurfaceCreated` sites pass
    `surface.owner_uid` from the freshly-created surface;
    the two `emitClientDisconnected` and two
    `emitSurfaceDestroyed` sites add a defensive lookup
    (`self.clients.sessions.get(...)` / `self.surfaces.getSurface(...)`)
    to capture the uid before the session or surface is
    deallocated, with `privilege.NOBODY_UID` as a fallback
    for the unexpected-already-removed case.

The bench-verification cycle was unusually instructive. The
first attempt failed because the bench-verification grep
ran while `semadraw-term` was still alive in the foreground
shell, so the disconnect events hadn't fired yet. The
second attempt failed because the binary in `/usr/local/bin`
predated the install run (the daemon was at 400s uptime,
not freshly restarted). The third attempt succeeded after
a clean `sudo sh install.sh` cycle, a healthy
`s6-svstat`-confirmed restart, and an actual
`semadraw-term --fullscreen` -> Ctrl+D cycle that exercised
all three observed events
(`client_connected`, `surface_created`, `client_disconnected`)
with `peer_uid:1001` and `owner_uid:1001`.

`surface_destroyed` did not appear in the bench output, but
that is pre-existing intentional behaviour: the
disconnect-time cleanup goes through
`removeClientSurfaces` which does not fire
`surface_destroyed` events. The event fires only when a
client explicitly sends `DESTROY_SURFACE` before
disconnecting; semadraw-term does not. Part B's code path
for `surface_destroyed` is exercised by Unix's explicit
DESTROY_SURFACE handler and TCP's equivalent, both of which
were updated with the defensive-lookup pattern, even though
the bench cycle did not visibly drive them tonight.

## What went wrong (second continuation)

### Bench-verify grep ran against a still-alive client

I gave Vic a bench-verification procedure for part B that
ran `grep client_disconnected` and `grep surface_destroyed`
without first instructing Vic to disconnect the
`semadraw-term`. Vic correctly noted "I wasn't worried,
just curious" when raising the empty-result question, but
the underlying issue was that my procedure missed the
disconnect-trigger step. The disconnect events couldn't
fire because no disconnect had occurred.

Lesson: bench-verification procedures need to be honest
about which side effects trigger which events. The Ctrl+D
in the inner shell of `semadraw-term` is what triggers the
disconnect event. Omitting that step made the procedure
unrunnable as written, and the empty grep result was a
correct response to an incomplete instruction set, not a
bug.

### The strings-grep diagnostic is unreliable

When the first bench-verify attempt came up empty, I
escalated to `strings /usr/local/bin/semadrawd | grep peer_uid`
as a diagnostic for "is the binary current?" That check
came back empty too, and I treated it as evidence the
binary was stale. It wasn't conclusive evidence; Zig's
format strings may not appear as contiguous byte runs that
`strings` can find. On the third attempt (the one that
succeeded), `strings | grep -c peer_uid` returned 2, which
disproved my caution about the technique for this case.

Lesson: `strings`-grep is useful as a sanity check but
should not be the *primary* test for "is the new code
deployed." The reliable test is observing the new field in
emitted output (which is what the bench cycle is supposed
to do directly).

### The build state was older than the source

The second-attempt failure ultimately turned out to be: the
daemon process had been running for 400 seconds (uptime
from `s6-svstat`), which meant it had not restarted during
the `sudo sh install.sh` Vic ran. install.sh's
`install_bin_required` hardening from earlier today catches
*missing* artifacts but does not catch *stale* artifacts
that didn't trigger a restart. The fix was a clean
`sudo sh install.sh` cycle with the daemon explicitly
restarted, which then ran the new binary.

Lesson: stale-running-process is a real failure mode for
the install.sh path even after install_bin_required. A
future install.sh hardening could explicitly invoke
`s6-svc -r` after binary copy to force a restart and assert
post-restart uptime is small. Out of scope tonight; worth
recording.
