## Architectural decisions reached

### vt(4) retirement is gated on a login replacement

The current AD-11 entry (`Console output: replace vt(4) for UTF
sessions`) is silent on the fact that vt(4) is the system's login
provider via `getty(8)` on `/dev/ttyv*`. Retiring vt(4) without a
replacement leaves the system unloggable-into at the console.

This is the third hat vt(4) wears, alongside framebuffer ownership
(what AD-4 wants) and console rendering (what AD-11's text claims).
The login hat is the one most easily missed.

### Plan 9 host-owner model considered and dropped

A no-login-prompt model in the Plan 9 style (single host owner
established at boot, no `getty`, capability-based mediation for
remote access via authfs) was discussed and seriously considered.
It was the cleanest expression of the substrate-replacement
principle. It is incompatible with multi-user-with-selection,
which is a hard requirement.

The Plan 9 framing also surfaced a subtler issue: it tried to push
desktop-layer decisions (identity, session) into UTF substrate
territory, which is not where they belong. UTF has no opinion about
users; it deals in uids from `getpeereid(3)`. The Plan 9 framing
felt awkward partly because of that layer confusion.

### Option Y: graphical login via PAM

Decision: build a UTF-native graphical login. vt(4) stays compiled
into the kernel as a recovery / single-user-mode fallback, but is
not part of the normal boot path under PGSD. drawfs takes the
framebuffer; a privileged daemon draws a login screen as a UTF
client; on auth success the daemon `setuid`s and execs the user's
session leader.

This is sddm/gdm/lightdm territory. It is real implementation work
(estimate: roughly comparable to AD-3 audiofs, smaller than AD-4
graphics replacement). It is load-bearing for AD-10 framebuffer
handoff: once drawfs takes the framebuffer at boot, something must
draw on it or the user sees a blank screen and has no way in.

#### Confirmed sub-decisions

- **Authentication backend**: PAM, `pam_unix` against
  `/etc/master.passwd`. Identical to what `login(1)` and `sshd` do.
  PAM stack at `/etc/pam.d/pgsd-sessiond` (or whatever the daemon's
  final binary name resolves to).
- **User enumeration**: all users in `/etc/master.passwd` with
  UID > 1000 and a shell present in `/etc/shells`. (Note: FreeBSD's
  `adduser(8)` starts new users at 1001 by convention, so
  `> 1000` excludes only UID 1000 itself, which on most installs
  is unset or holds an admin/root-equivalent. Worth confirming in
  the ADR whether `>=` was intended; current memo records `>` per
  the explicit answer given.)
- **Architecture**: single system-wide semadrawd running as a
  dedicated `_semadraw` system user (not root), accepting
  connections from any user authenticated via local socket
  `getpeereid`. Per-user surface namespacing and event routing by
  ownership. The login daemon is itself a privileged semadrawd
  client.

#### Open sub-decisions

- **UID convention**: `> 1000` vs `>= 1000`. Documentation needed.
- **Login screen v1 visual scope**: user list only, or list +
  password panel + shutdown/reboot buttons + session-type picker?
  Recommendation pending: list with password overlay, shutdown
  buttons present, no avatars in v1, no session picker in v1
  (one default session per user, configurable in user config).
- **Per-user attribute file mechanism in v1 vs deferred**: a
  parallel `/etc/utf/users/<name>.conf` (or similar) for
  PGSD-specific per-user metadata (display name, session, eventual
  age bracket, capability flags). In-v1 means more work upfront
  but coherent design; deferred means simpler v1 but a rewrite
  later. Not yet decided.

### Age attestation is out of scope for UTF/PGSD compliance

Surveyed the regulatory landscape: California AB-1043 (effective
January 2027, signed October 2025), Colorado SB 26-051, federal HR
8250 (Parents Decide Act, in committee as of April 2026), similar
pending in CO/IL/LA/NY/TX/UT.

Verified mechanism is self-attestation. There is no actual
verification; users state an age, the OS stores it, apps query it
via a platform API. The "verification" is operator/installer
honour-system.

PGSDF is a Tokyo-based foundation building a small open-source
FreeBSD distribution. It has no US legal nexus comparable to
Apple/Google/Microsoft and is in the same position as Arch, Debian,
Gentoo: no centralized account infrastructure, no legal team, no
realistic compliance path. The FOSS-world response (System76,
Canonical, Fedora discussions) is largely "disclaim and document,
do not implement."

PGSD adopts the same posture: no compliance machinery; document the
stance in `docs/POLICY.md` (to be drafted).

A separate question, *not* compliance-driven, is whether the
session-manager should expose a generic per-user attribute mechanism
that operators can use for kiosk modes, parental controls, or
jurisdictional compliance under their own policy. This is a useful
capability independent of the law and is what the "Per-user
attribute file mechanism" open question above is pointing at.

### Substrate vs distribution: architectural split

UTF is the substrate (kernel modules and userland services
providing a unified temporal fabric for input, audio, graphics,
time). PGSD is one distribution built on UTF. They live in the
same monorepo for development convenience but they are
architecturally distinct.

UTF userland convention: `sema-` prefix (`semadrawd`, `semaaud`,
`semainput`).

PGSD distribution-layer convention: `pgsd-` prefix
(`pgsd-sessiond`, future `pgsd-*d` daemons).

This naming is self-documenting: a process in `ps` listings
immediately tells the operator which layer it belongs to. The
substrate has stable contracts (`/var/run/sema/` regions, kernel
module ABIs, IPC protocol); a different distribution could in
principle adopt UTF and build its own desktop on top.

Practical implication: README needs to make this split visible.
BACKLOG categorisation should reflect it (AD-* substrate work,
NDE-* PGSD desktop, LT-* layer tree, new SM-* session
management track).

### Session-manager is desktop-agnostic

Decision: the session-manager is its own track, a peer to NDE and
LT, not a sub-item of NDE. It does not encode any NDE-specific
behaviour. Session types are first-class: `.session` files under
`/usr/local/share/pgsd/sessions/` enumerate the available
environments (NDE, single-fullscreen-`semadraw-term`,
custom-application kiosk, eventually X11 sessions via NDE-5).
The session-manager authenticates the user and execs the chosen
session leader.

Daemon name: `pgsd-sessiond`.

Track code letter: SM (Session Management). Future SM items might
cover screen lock, idle handling, fast user switching,
suspend/resume, agent-of-record for `XDG_RUNTIME_DIR`-equivalent.

This reframing makes a few scenarios first-class that were
previously implicit:

- PGSD with no NDE: log in, drop into fullscreen `semadraw-term`,
  done. Useful as a minimal-install or recovery configuration.
- PGSD with X11 only (post-NDE-5): session-manager launches an
  X session against the X11 bridge.
- Dedicated visualization workstation: boot directly into a
  custom radar-visualization application full-screen, no general-
  purpose desktop. This is plausibly PGSD's primary use case
  given PGSDF's focus on METOC and scientific visualization.

## Coda: resolutions reached after initial draft

The five open questions enumerated above were addressed at the
end of the same session, before the memo was committed. Recorded
here for the next session's planning rather than re-edited above,
so the original open-then-resolved arc is preserved.

### 1. UID convention

**`UID > 1000`**, not `>= 1000`. Confirmed as the FreeBSD
convention preferred for this project. UID 1000 itself is
excluded from the user picker (typically unset or held by an
administrative account).

User-enumeration query for `pgsd-sessiond`:

    for entry in /etc/master.passwd:
        if pw_uid > 1000 and pw_shell in /etc/shells:
            include

### 2. Per-user attribute file in v1

**In v1.** `/etc/utf/users/<username>.conf` (final path TBD in
the SM-1 ADR) ships in the first version of `pgsd-sessiond`.
Coherent design from the start; slightly bigger initial
implementation accepted as the cost.

Fields the ADR is owed:

  - `display_name`: UI override of GECOS.
  - `default_session`: name of the `.session` file to launch by
    default for this user.
  - `avatar_path`: reserved field. Login UI defers rendering to
    v2; the field exists in v1 so the file format is stable.
  - `age_bracket`: one of `under-13`, `13-15`, `16-17`, `adult`,
    `unspecified`. Default `unspecified`. Operator-set, not
    user-set. Not exposed to applications via any system API.
    Not subject to compliance attestation by PGSDF.
  - `capabilities`: list of capability flag strings
    (`can-shutdown`, `can-add-users`, future entries). Where
    possible mapped to FreeBSD groups (`wheel`, `operator`) via
    PAM session hooks.

Format and behaviour the ADR is owed:

  - File format: simple `key = value` pairs, one field per
    line, comments with `#`.
  - Missing fields: handled with sane defaults, no crash.
  - Creation: manual editing in v1. A `pgsd-useradd` wrapper is
    future work, not v1 scope.
  - Read strategy: read-on-login, not file-watch. Simpler and
    correct for v1.

### 3. Login screen v1 visual scope

  - **Avatars in v1: no.** Defer rendering to v2. Reserve the
    `avatar_path` field per above.
  - **Shutdown buttons in v1: yes.** Power off, restart, and
    suspend (where the platform supports it; FreeBSD has
    `acpiconf -s 3`). Locked-out users must be able to power
    down without authenticating. Standard placement: bottom-
    right of the login screen.
  - **Session picker exposed in v1: yes.** Once a user is
    selected and before the password is entered, a dropdown
    next to or below the user name shows the current default
    session and permits an explicit override. Override applies
    to that login only; does not write back to
    `default_session` in the user's attribute file.

This locks the v1 login UI shape: vertical list of users
sorted by display name (GECOS or `display_name` override) with
fallback to username, tap-a-user to expand a password field in
an overlay, session dropdown adjacent to the user name,
shutdown / restart / suspend buttons in a corner. No avatar
column. No "Other..." escape hatch (only enrolled users
appear; admin accounts under UID 1000 are not in the picker).

### 4. `docs/POLICY.md`

**Yes, write the document.** Technical tone. Position implicit,
not stated. Same posture FreeBSD takes toward most regulatory
regimes: describe what the system does and does not do, do not
editorialize on whether regulators are right or wrong.

Sections the document will cover:

  - PGSD account model: how users are enrolled
    (`/etc/master.passwd`, `pgsd-sessiond` enumeration via
    `UID > 1000` and `/etc/shells`), what attributes are
    stored (`/etc/utf/users/<name>.conf`), what is and is not
    validated by PGSD.
  - Self-reported attributes: notes that `age_bracket` and
    similar fields are operator-set, not validated by PGSD,
    not exposed to applications via any system API, and not
    subject to compliance attestation by PGSDF.
  - Jurisdictional notes: factual list of regulatory
    frameworks (California AB-1043, Colorado SB 26-051,
    federal HR 8250 in committee, similar pending in
    CO/IL/LA/NY/TX/UT). Statement that PGSD does not
    implement compliance machinery for these frameworks.
  - Operator responsibilities: if an operator wishes to
    deploy PGSD in a context with regulatory requirements,
    the per-user attribute mechanism may be used as a
    building block, but compliance is the operator's
    responsibility, not PGSDF's.

Language to avoid: any phrasing that takes a position
("PGSDF believes", "we object to", "the law is poorly
designed", or analogous wording in the other direction).
Just: here is what PGSD does, here is what PGSD does not do,
here is where the responsibility line falls.

### 5. drawcons design as its own ADR before AD-11 reopens

**Confirmed.** drawcons is not a small patch on AD-11; it is
its own design effort and owes its own ADR before AD-11 can
be reopened.

ADR will cover:

  - Path: probably `drawfs/docs/adr/0001-drawcons-design.md`
    or similar, alongside other drawfs design notes.
  - Kernel-side font: likely the existing 8x16 font already
    embedded for the eventual console path. Confirm in the
    ADR rather than re-derive.
  - `cnputs` implementation: registers drawcons as a `cnX`
    device with FreeBSD's console framework.
  - Framebuffer ownership during panic: drawcons takes the
    framebuffer back from any user-space owner during panic,
    even if semadrawd or `pgsd-sessiond` was holding it.
  - Scrollback: probably none in v1, possibly a fixed ring
    buffer of N lines. ADR decides.
  - `boot -s` interaction: single-user mode probably needs
    full drawcons functionality (interactive shell input and
    output) rather than panic-output only. ADR scopes both.

Out of scope for the next session. Owed before AD-11 is
reopened, which is far future. The AD-11 BACKLOG entry's
reframing in the next-session reshape will cite drawcons-ADR
as a prerequisite.
