### Deviation flagged

`XDG_RUNTIME_DIR=/var/run/xdg/vic` does NOT equal
`PGSD_RUNTIME_DIR=/var/run/pgsd/1001`. ADR 0005 line 146 mandates
they be equal. The deviation is coming from either login.conf
(via LOGIN_SETENV) or a PAM module (via pam_getenvlist) - our
base env DOES set XDG_RUNTIME_DIR to the same value as
PGSD_RUNTIME_DIR, but something downstream overrides it.

The fix is straightforward: after applying PAM env in
`childPath`, explicitly re-set XDG_RUNTIME_DIR and PGSD_RUNTIME_DIR
so the ADR 0005 invariant always holds regardless of what
login.conf or PAM contribute. Filed as a follow-up rather than
folded into Stage 4 because it's a real behavior change deserving
its own commit and a re-bench.

## pkg upgrade incident

Mid-session, `pkg upgrade` on the bare-metal bench was run. Side
effect: `/boot/kernel/` was overwritten with the stock FreeBSD
GENERIC kernel, replacing the previously-installed PGSD kernel.
The custom kernel was rebuilt and reinstalled via
`pgsd-kernel-build.sh build` + `install`, using the existing
WITHOUT_MODULES discipline from AD-8.

### Structural follow-up flagged but not done

The root cause is that `pgsd-kernel-build.sh install` writes to
`/boot/kernel/`, the same path the FreeBSD-kernel-generic pkgbase
package owns. Every kernel-touching `pkg upgrade` will clobber
PGSD's kernel until the install destination changes.

Recommended fix (deferred to its own commit and ADR):

  - Change `pgsd-kernel-build.sh` to use `INSTKERNNAME=kernel.PGSD`
    so the kernel installs to `/boot/kernel.PGSD/` alongside its
    own modules.
  - Add `/boot/loader.conf` entry `kernel="kernel.PGSD"` so the
    bootloader picks PGSD by default.
  - Update the AD-8 closure verification in `pgsd-kernel-build.sh`
    to look in the parameterised install path rather than
    `/boot/kernel/`.
  - GENERIC stays at `/boot/kernel/` as a fallback recoverable
    via `nextboot -k kernel`.

### Refactor: session resolution moved up to runLaunch

Before Stage 4, `launch_mod.launch` took a session name and
called `resolveSessionExec` internally to find the `.session`
file. After Stage 4, `runLaunch` in main.zig calls
`session_file.lookupById` BEFORE authenticating, then passes the
resolved id and exec value into `launch_mod.launch`. The motive
is user-experience: typing the wrong session name should not cost
the user a password entry.

This is what enables Test 5 below.
