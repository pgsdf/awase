# 2026-05-16

Working session on `pgsd-sessiond` covering Stage 5's fullscreen +
3x font scale rework, the Stage 6 wiring of PAM auth and session
launch into the graphical login UI, two build-fix iterations, three
config issues caught during bench, and end-to-end bench verification
on `pgsd-bare-metal`.

Stages 5 and 6 complete: `pgsd-sessiond --ui-only` now performs a
full graphical login flow on PGSD, from amber-on-black "Identify:"
prompt through PAM authentication to fork+exec of the user's
session leader.

## Stage 5 fullscreen rework

Stage 5's initial landing (`f8e0748` + the two fingerprint fixes
`1c37f92` / `b56e319`) drew a fixed 1024x768 surface at the
glyph atlas's native 8x16 cell size. On a 3840x2160 iMac that
appeared as a small unscaled window in the corner of the display.

Reworked in two coordinated commits to match what
`semadraw-term --fullscreen --scale 3` does:

- `bdf47d4 semadraw: re-export low-level client module`. One-line
  change to `semadraw/src/semadraw.zig` adding
  `pub const client = @import("semadraw_client");` so downstream
  packages can access `Connection`, `Surface`, and `Event`
  directly. The `App` framework creates its surface at fixed
  dimensions during `App.init`; sizing the surface to the
  framebuffer requires calling `conn.queryOutputInfo(0)` BEFORE
  surface creation, which `App` doesn't permit.

- `0e704ed pgsd-sessiond: fullscreen login UI with 3x font scale`.
  Dropped `App` from `runUiOnly` in favor of a direct
  `Connection` + `Surface` event loop modeled on
  `semadraw-term`'s `main.zig`. Surface is created at the
  framebuffer dimensions returned by `queryOutputInfo`, with a
  fallback to `ui.FALLBACK_WIDTH/FALLBACK_HEIGHT` (1024x768) if
  the daemon refuses or has no output ready. Text rendering
  passes `cell_width = GLYPH_WIDTH * SCALE` (3) to `drawGlyphRun`,
  the same approach term uses for HiDPI; all layout constants in
  `ui.zig` multiply by SCALE in draw paths. Layout centers on the
  queried surface dimensions, not on a hardcoded size.

Bench-verified on `pgsd-bare-metal`. stderr printed:

```
info(semadraw_client): connected to semadrawd, client_id=3
fullscreen: framebuffer 3840x2160 at scale 3
```

The login surface filled the iMac's display with 24x48 px
characters and the layout centered correctly. Vic confirmed
"looks great".

## Stage 6 wiring

Two coordinated commits that wire PAM authentication and session
launch into the previously-stub `--ui-only` mode:

- `169c6a7 semadraw: set CLOEXEC on client connection socket`.
  Added `posix.SOCK.CLOEXEC` to the `posix.socket()` call in
  `Connection.doConnect`. Every other socket creation in semadraw
  already uses `CLOEXEC` (remote_connection, socket_server,
  shm.memfd_create); the local Unix-domain client connection was
  the lone outlier and predated the convention. Stage 6's
  fork+exec path is the first consumer that would have leaked the
  fd. Belt-and-suspenders with the explicit `conn.disconnect()`
  that pgsd-sessiond now does before invoking `launch()`.

- `a652aa1 pgsd-sessiond: implement stage 6 (full graphical login
  flow)`. 591 lines net across `main.zig` and `ui.zig`. Major
  components:

  - **`BufferedConv`**: a new `pam.Conversation` impl parallel to
    `CliConv`, serving the pre-collected username and password
    from `state.username.items` and `state.password.items` rather
    than reading from a TTY. Allocates response strings with
    `std.c.malloc` so libpam's `free(3)` matches.

  - **`ui.State.submitting`**: a third `FieldState` value, entered
    when the user presses Enter on a non-empty password. The UI
    locks input (`handleAction` returns early except for Ctrl-Q)
    while `main.runUiOnly` performs PAM auth on the next loop
    iteration. `draw()` renders an "Authenticating..." indicator
    in amber between the password field and the hint line.

  - **`ui.State.status_message`**: a new `?[]const u8` field
    rendered in amber above the hint line. `main` formats failure
    messages into a stack-allocated `[256]u8` and passes a slice
    to `state.resetForRetry(msg)`, which also clears the password
    buffer and returns the field to `.password` with username
    preserved (standard X login screen behavior). The message is
    cleared automatically on the next print/backspace/clear so it
    disappears as soon as the user reacts.

  - **`attemptAuth`**: orchestrates one
    user-lookup ã `pam.start` ã `authenticate` ã `acctMgmt` ã
    session-resolve cycle, returning a tagged `AuthOutcome` of
    `.success` / `.retry` / `.fatal`. `AuthFailed` and
    `UserUnknown` both produce the same masked
    "authentication failed; N attempts remaining" so the UI
    doesn't leak which usernames exist. Resource cleanup on the
    retry/fatal paths uses explicit owned-flag `defer` rather
    than `errdefer` because the function returns
    `AuthOutcome` not `!T`; the success path clears the flags so
    the caller takes ownership of `user` / `session` / `transaction`.

  - **Session resolution chain**: `user.attrs.default_session`
    (from `/etc/utf/users/<name>.conf`) is tried first, falling
    back to system `"default"` if absent. Resolves through
    `session_file.lookupById` against
    `/usr/local/share/pgsd/sessions/`.

  - **Surface teardown before exec**: on auth success, `runUiOnly`
    explicitly calls `surface.destroy()` and `conn.disconnect()`
    BEFORE invoking `launch_mod.launch`. The child semadraw-term
    opens its own fresh connection (visible in the bench output
    as `client_id=4`, distinct from pgsd-sessiond's `client_id=3`).

  - **Root check**: `--ui-only` now requires root (`geteuid() == 0`)
    just like `--launch` does, because PAM session-open, setusercontext,
    and the `/var/run/pgsd/<uid>/` chown all need it. Fails fast
    with `EXIT_USAGE` if not root.

  - **Retry policy**: 3 attempts total, per the design Q&A in
    the session. Third failure exits with `EXIT_AUTH_FAIL` (1).

  - **8 new ui.zig unit tests** covering the `.submitting`
    transition, input lock during submission, `resetForRetry`
    semantics, and `status_message` clearing on input.

## Build-fix iterations

### Stage 5 fingerprints

The Stage 5 initial commits (`f8e0748` + the semadraw companion)
used build.zig.zon fingerprints generated as a deterministic
sha256 of the package name. Zig 0.15 rejected both:

```
build.zig.zon:1:2: error: invalid fingerprint: 0x...; if this is
a new or forked package, use this value: 0x...
```

Zig fingerprints have the upper 32 bits as a name-derived
checksum and the lower 32 bits as a random id; the deterministic
scheme didn't match Zig's validation. Two fix commits (`1c37f92`,
`b56e319`) replaced the fingerprints with the values Zig reported.
Permanent for the package per the build.zig.zon documentation.

### Stage 6 switch-capture mutability

After the Stage 6 patch applied cleanly, `zig build` failed with:

```
src/main.zig:846:50: error: expected type '*main.ReadyToLaunch',
found '*const main.ReadyToLaunch'
                    final_exit = doLaunch(alloc, s);
                                                 ^
```

The switch arm captured the `.success` payload as `|*s|` to get
a `*ReadyToLaunch`. But the value being switched on
(`const outcome = attemptAuth(...)`) was const, so the pointer
capture yielded `*const ReadyToLaunch`. `doLaunch`, `user.deinit`,
`session.deinit`, and `transaction.end` all need mutable
receivers.

Fix in `155caeb`: capture by value (`|s_const|`), materialize a
mutable local (`var s = s_const`), pass `&s` to `doLaunch`. The
struct copy aliases the heap pointers inside but that's fine; we
never read `s_const` again.

### Three config issues caught during bench

These were not code bugs; they were missing or incorrect setup
on `pgsd-bare-metal`.

1. **`default_session = semadraw-term` in `/etc/utf/users/vic.conf`**.
   Session IDs are .session file basenames, not exec names; this
   resolved to a missing `semadraw-term.session`. Fix: edit to
   `default_session = default` (or remove the line; Stage 6 falls
   back to `"default"`).

2. **`/usr/local/share/pgsd/sessions/` did not exist on `pgsd-bare-metal`**.
   The Stage 3 bench installer hadn't been run there. Fix:
   `doas sh install-bench.sh`.

3. **`/etc/pam.d/pgsd-sessiond` not installed on `pgsd-bare-metal`**.
   Same root cause as #2. Stage 6's `selectService` correctly fell
   back to the `login` service per Stage 1's logic; the bench
   succeeded under the fallback, but `install-bench.sh` should be
   run on every PGSD host that runs `pgsd-sessiond`.

# 2026-05-16 (later session): Stages 7-9 and the inputfs-ring boot race

The earlier 2026-05-16 session above closed Stages 5 and 6. This

that ended in finding and fixing a startup race that left the
boot login screen keyboard-dead. It is recorded here in full,
including the wrong turns, because the wrong turns are the
useful part: the same misreadings are easy to repeat.

## The diagnostic arc, including the wrong turns

The reported symptom after Stage 9a deployment: "I rebooted and
could not enter text at the login prompt." The path from there
to the actual root cause took several wrong turns.

1. **AD-10 black-screen incident.** An early attempt to apply
   AD-10's loader.conf framebuffer-takeover tunables, before
   verifying drawfs.ko was installable as a boot module, black-
   screened the machine. Recovered. Lesson, now also written
   into the AD-10 ADR's Risks section: drawfs.ko boot-module
   installability must be proven before any loader.conf tunable
   that affects the console goes anywhere near a reboot. AD-10
   implementation remains gated on that.

2. **Misread shutdown logs.** pgsd-sessiond's log contained
   `shutdown: [pid NNNN]` / `Shutdown NOW!` lines. These were
   read as evidence that pgsd-sessiond was driving the machine
   into a shutdown loop, which produced a whole theory about
   the Stage 8 power menu firing on phantom input at boot. It
   was wrong: s6-log's `current` is a rolling multi-run file,
   and those lines were from an earlier manual Stage 8 power-
   menu test, not the boot under investigation. Lesson: a flat
   s6-log `current` is not a single-boot timeline; correlate by
   timestamp before theorizing.

3. **Two sigaction build failures.** The Stage 9a SIGTERM
   handler used `posix.empty_sigset` (does not exist in
   std.posix; only in std.os.linux) and `catch {}` on a
   void-returning `posix.sigaction`. Both were avoidable: the
   correct idiom (`std.mem.zeroes(posix.sigset_t)`, no catch)
   was already in use in `semadraw/src/daemon/semadrawd.zig`'s
   SIGPIPE handler. Lesson, repeated from prior sessions: grep
   the existing tree for a working use of an API before
   writing a new one from memory.

4. **The actual bug, found via a read-only diagnostic script.**
   A diagnostic capture on the booted bench showed: inputfs.ko
   loaded; four inputfs HID devices attached including the
   Apple keyboard; `/var/run/sema/input/events` present,
   `root:_semadraw 0640`, readable by `_semadraw` (semadrawd's
   dropped uid); rcorder correct (inputfs before semadraw); and
   yet semadrawd logging `inputfs ring at
   /var/run/sema/input/events unavailable; no input from
   inputfs`. Every "obvious" cause (absent file, tmpfs
   shadowing, permission mismatch, wrong boot order) was
   eliminated by evidence rather than argued away.

## Root cause: two independent bugs

**Bug A (the cause of the dead keyboard).**
`InputfsInput.init()` was a one-shot probe called once at
drawfs backend init. inputfs publishes its event ring
asynchronously during staged HID attach. At cold boot semadrawd
and inputfs come up near-simultaneously under s6; semadrawd's
single probe can run before inputfs has published the ring.
semadrawd then stored `inputfs = null` permanently and ran for
its whole lifetime with no input, so every client (the
pgsd-sessiond login screen) was keyboard-dead. Never reproduced
interactively because by the time semadrawd was started by hand
the ring had long since settled. Fix (`5e17071`): a `quiet`
parameter on `init()` so a retry path can probe without log
spam, plus a throttled re-probe in `pollEventsImpl` (every
`INPUTFS_RETRY_INTERVAL` = 120 frames) while `inputfs == null`,
latching permanently on success. The existing skip-to-head
(`reader.last_consumed = reader.writerSeq()`) makes a late latch
safe: no replay of a backlog of stale keystrokes. Steady-state
cost once latched is one counter compare per frame.

**Bug B (a latent correctness bug, and a blocker for the
rc.d-gate alternative).** The inputfs rc.d service and two
install-time checks used `kldstat -q -m inputfs`.
`DRIVER_MODULE(inputfs, hidbus, ...)` registers the module as
`hidbus/inputfs`, so `-m inputfs` never matched even with
inputfs.ko loaded. `service inputfs status` reported "not
loaded" while kldstat showed it loaded; the rc.d start guard
was always false (non-idempotent kldload); and the install-time
`INPUTFS_WAS_LOADED` probe never set, so the post-install
module-refresh silently never ran. Fix (`451b642`): all five
predicates use `kldstat -q -n inputfs.ko` (match the kld file,
stable regardless of internal module naming). Sequenced first
so `service inputfs status` was trustworthy while bench-
verifying Bug A.
